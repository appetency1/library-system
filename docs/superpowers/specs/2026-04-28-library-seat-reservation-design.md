# 图书馆座位预约系统设计规格

## 1. 项目目标

本项目实现一个前后端分离的图书馆座位预约系统，面向高校图书馆的座位管理和学生自助预约场景。系统需要支持学生注册登录、查询阅览室与座位状态、预约座位、取消预约、签到签退，以及管理员对阅览室、座位、用户、预约和公告的管理。

该作品用于计算机专业毕业设计展示，重点体现前端工程化、后端 REST API、数据库建模、权限控制、业务约束、数据统计与可运行部署。

## 2. 技术架构

### 2.1 前端

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- Axios
- ECharts
- CSS 变量与组件化样式

前端采用单页应用结构，按学生端和管理员端划分路由。登录后根据用户角色进入对应工作台。前端负责页面渲染、表单校验、状态管理、接口调用和图表展示。

### 2.2 后端

- Node.js
- Express
- TypeScript
- SQLite
- JSON Web Token
- bcrypt 密码哈希

后端提供 REST API，负责身份认证、权限校验、业务规则校验、数据库读写和统计数据聚合。后端不直接渲染页面，仅返回 JSON 数据。

### 2.3 数据库

数据库使用 SQLite，适合毕业设计本地演示和轻量部署。系统启动时提供初始化脚本，创建基础表结构并写入演示数据。

## 3. 角色与权限

### 3.1 学生

学生可以：

- 注册和登录
- 查看公告
- 查看阅览室列表
- 查看指定日期和时间段的座位状态
- 预约可用座位
- 查看个人预约记录
- 取消未开始预约
- 对已到时间的预约签到
- 对已签到预约签退

### 3.2 管理员

管理员可以：

- 登录后台
- 查看统计看板
- 管理用户状态
- 新增、编辑、停用阅览室
- 新增、编辑、停用座位
- 查看和筛选预约记录
- 强制取消异常预约
- 发布和管理公告

管理员不能替学生伪造签到签退记录，只能取消异常预约，以保持业务数据可信。

## 4. 核心业务流程

### 4.1 学生预约流程

1. 学生登录系统。
2. 选择阅览室、日期和时间段。
3. 系统返回座位状态，包含可预约、已预约、停用三类状态。
4. 学生选择可用座位并提交预约。
5. 后端校验用户、座位、时间段和冲突规则。
6. 预约成功后生成预约记录，状态为 `reserved`。

### 4.2 签到签退流程

1. 到达预约开始时间后，学生可对预约签到。
2. 签到成功后，预约状态变为 `checked_in`，记录签到时间。
3. 学生离馆时签退，预约状态变为 `completed`，记录签退时间。
4. 未签到且已过预约结束时间的记录可由系统查询时标记为 `expired`，或由管理员在预约管理中处理。

### 4.3 管理端维护流程

1. 管理员登录后台。
2. 管理阅览室和座位基础信息。
3. 查看预约记录和用户状态。
4. 对违规、异常或维护场景下的预约执行强制取消。
5. 通过统计看板查看今日预约数、签到率、座位使用率和近 7 日预约趋势。

## 5. 业务规则

- 同一用户在同一日期、同一时间段只能拥有一个有效预约。
- 同一座位在同一日期、同一时间段只能被一个有效预约占用。
- 已停用座位不能被预约。
- 已停用阅览室下的座位不能被预约。
- 预约开始前允许学生取消。
- 预约开始后不允许学生取消，只能签到、签退或等待过期处理。
- 只有预约本人可以签到和签退。
- 管理员可以取消 `reserved` 或 `checked_in` 状态的预约。
- 用户被禁用后不能登录，也不能创建新预约。

## 6. 数据模型

### 6.1 users

- `id`: 主键
- `username`: 登录账号，唯一
- `password_hash`: 密码哈希
- `name`: 姓名
- `role`: `student` 或 `admin`
- `status`: `active` 或 `disabled`
- `created_at`: 创建时间

### 6.2 rooms

- `id`: 主键
- `name`: 阅览室名称
- `location`: 位置
- `description`: 描述
- `status`: `active` 或 `disabled`
- `created_at`: 创建时间

### 6.3 seats

- `id`: 主键
- `room_id`: 所属阅览室
- `seat_no`: 座位编号
- `type`: 普通座、靠窗座、电源座等
- `status`: `active` 或 `disabled`
- `created_at`: 创建时间

### 6.4 reservations

- `id`: 主键
- `user_id`: 预约用户
- `room_id`: 阅览室
- `seat_id`: 座位
- `reserve_date`: 预约日期
- `start_time`: 开始时间
- `end_time`: 结束时间
- `status`: `reserved`、`checked_in`、`completed`、`cancelled`、`expired`
- `checkin_at`: 签到时间
- `checkout_at`: 签退时间
- `cancelled_at`: 取消时间
- `created_at`: 创建时间

### 6.5 notices

- `id`: 主键
- `title`: 标题
- `content`: 内容
- `status`: `published` 或 `draft`
- `created_at`: 创建时间

## 7. API 设计

### 7.1 认证

- `POST /api/auth/register`: 学生注册
- `POST /api/auth/login`: 用户登录
- `GET /api/auth/me`: 获取当前登录用户

### 7.2 学生端

- `GET /api/notices`: 公告列表
- `GET /api/rooms`: 阅览室列表
- `GET /api/seats`: 按阅览室、日期和时间段查询座位状态
- `POST /api/reservations`: 创建预约
- `GET /api/my/reservations`: 我的预约
- `POST /api/reservations/:id/cancel`: 取消预约
- `POST /api/reservations/:id/checkin`: 签到
- `POST /api/reservations/:id/checkout`: 签退

### 7.3 管理端

- `GET /api/admin/stats`: 统计看板数据
- `GET /api/admin/users`: 用户列表
- `PATCH /api/admin/users/:id/status`: 修改用户状态
- `POST /api/admin/rooms`: 新增阅览室
- `PATCH /api/admin/rooms/:id`: 编辑阅览室
- `POST /api/admin/seats`: 新增座位
- `PATCH /api/admin/seats/:id`: 编辑座位
- `GET /api/admin/reservations`: 预约管理列表
- `POST /api/admin/reservations/:id/cancel`: 管理员取消预约
- `POST /api/admin/notices`: 发布公告
- `PATCH /api/admin/notices/:id`: 编辑公告

### 8.1 公共页面

- 公告管理：发布、编辑和上下线公告

## 9. 错误处理

后端统一返回 JSON 错误结构：

```json
{
  "message": "错误说明"
}
```

常见错误包括：

- 登录失败
- Token 失效
- 权限不足
- 参数缺失或格式错误
- 预约冲突
- 座位或阅览室已停用
- 预约状态不允许当前操作

前端通过统一 Axios 拦截器处理登录过期、权限错误和通用提示。
